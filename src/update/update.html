Modern. Safe. Performant. The complete guide to the Vix programming language.

Introduction

Vix is a modern programming language designed to balance performance, memory safety, and developer productivity. It combines the speed of C with the safety features of modern languages, making it suitable for both low-level system programming and high-level application development.

Current Status: Alpha version. Expect breaking changes as we rapidly iterate towards stability.

Key Features

  • Performance: Compiles to C, then to LLVM using Clang. Achieves C-level performance with SSA optimizations.
  • Safety: Immutable variables by default, no null (uses None), built-in Option and Result types.
  • Developer Experience: Simple syntax using end keyword instead of braces. Beginner-friendly design.
  • Interoperability: Seamless integration with C/C++ libraries.

    Interpreter VM

    Vix is known as a compiled language that generates C IR and uses clang for compiling to machine code. However, this process can be slow. If you want to fix bugs/errors quickly without waiting for lengthy compilation every time, you can use the Vix Interpreted VM. It doesn't generate compiled applications like .exe files—it simply runs your code inside a VM (interpreted), using LLVM and clang to make this possible. This implementation is still in the planning and prototype stages, and there's no guarantee it's 100% coming. If you want to help us make this possible, let us know in "issues".

    // this function has "Result" and can return "ok" or "err". func some_function(input: Option[str]): Result[T: str, E: str] input.unwrap_or( return Err("Failed to unwrap string aka: string is a none") ) Ok(input) end # When we send "None" it will return an error if Ok(output) = some_function(None) then end print("No! function is not okay, it's returning an error") if Err(err) = some_function(None) then print("We got it the error is", err) end

    But functions can return None. You can fix this by:

    func other_function(): Result[Option[int32], str] return Ok(None) end if Some(output) = other_function() then # ... end

    If you want to just get the output:

    func example_function(): int32 return 10 end if let(output) = example_function() then print(output) end

    The full update added let(), Some(), Err(), and Ok() for the "if" statement for more safety and better memory management. You can ignore the error or output using "_" like this:

    if Ok(_) = example_function() then print("function is successful") end if Err(_) = example_function() then print("function just failed") end

    Deep Explanation & Examples

    .as_bytes() - Converting to Byte Slices

    Vix uses this to convert string data into a slice of bytes (&[u8]), providing a view of the underlying UTF-8 encoded data. It's useful for network programming (which requires sending raw bytes), I/O operations that work with raw bytes, and cryptographic operations (hashing, encryption), and much more.

    create txt = "Hello" bytes = txt.as_bytes() # &[u8] = [72, 101, 108, 108, 111]

    Strings in Vix are always UTF-8. as_bytes gives you:

    • Safe, zero-cost view of that UTF-8 data
    • Non-copying with no overhead—you're just viewing the same memory with a different type
    • Returns an immutable reference (&[u8])

    .as_ptr() - Getting a Raw Immutable Pointer

    Vix uses this to return a raw pointer (*const T) to the first element of the data, giving you direct memory access. Common usage includes: FFI with C/C++, passing data to system calls, low-level memory operations, and performance-critical code where you need to bypass bounds checking.

    create numbers = [1, 2, 3, 4, 5] create ptr: *const int32 = numbers.as_ptr(); unsafe: print("First element:", *ptr) end
    • Returns *const T (immutable raw pointer)
    • The pointer points to the first element in the data
    • The pointer is only valid while the original data exists
    Warning:
    • Requires unsafe blocks to run and use
    • If the pointer is empty, you'll get a non-null dangling pointer that can easily crash your program or corrupt your memory!
    • Ensure the data isn't moved or dropped
    • Violating aliasing rules (multiple mutable accesses) is undefined behavior

    .as_mut_ptr() - Getting a Raw Mutable Pointer

    Vix uses this to return a raw mutable pointer (*mut T), allowing direct memory modification without Rust's borrow checker oversight. Like as_ptr(), it has mostly the same usage in: FFI with C, implementing custom data structures, performance-critical code with manual memory management, and interfacing with hardware or memory-mapped I/O.

    create mut numbers = [1, 2, 3, 4, 5] create ptr: *mut int32 = numbers.as_mut_ptr(); unsafe: *ptr = 100 *ptr.slot(1) = 200 end print(numbers)
    • Returns *mut T (mutable raw pointer)
    • Requires mutable access to the original data (.as_mut_ptr() needs &mut self)
    Warning:
    • More dangerous than .as_ptr() because it allows modification
    • No aliasing: Don't create multiple mutable references to the same memory
    • Bounds checking: You're responsible for staying within valid memory

    Comparison Table

    Function Returns Mutability Safety Common Use Case
    .as_bytes() &[u8] Immutable Safe String → bytes conversion
    .as_ptr() *const T Immutable Unsafe FFI, reading raw memory
    .as_mut_ptr() *mut T Mutable Unsafe FFI, modifying raw memory

    Update Alpha 3.25

    This update contains fixes for 3 main bugs and 5 additional bugs that were making Vix unusable and could lead to problems in the future. Special thanks to nevakrien for reporting critical problems in Vix!

    Bugs Fixed

    First Main Bug: Empty Tuple Generation

    The first major bug in "Tuple" generation was not allowing () (empty tuple). This caused problems. When you created an empty tuple before this bug was fixed, it would always generate:

    typedef struct { void field_0; // "field_0" is forbidden "void" Slice_char field_1; } Tuple_void_char8;

    This was incorrect. We have fixed this bug and now it allows empty tuples ():

    func example(): Result[(), str] Ok() # Now works fine! Err("some err") # Works too! end

    Second Main Bug: Malloc Null Pointer Issue

    The second main bug is a really dangerous one because it can lead to your program exhibiting random behavior. We use malloc() for many things, from built-in functions to features inside the language. Thanks to nevakrien for finding this bug.

    Malloc can return null, which will cause many problems like random behavior, crashes, etc. We have fixed this by adding a checker function to ensure it's not null:

    void* vix_alloc(size_t s){ void* ans = malloc(s); // we going to add better error handling in future. But for now this is the function to at least remove this problem if(!ans) exit(1); return ans; }

    Third Main Bug: Enum Field Syntax

    Enums previously could not generate Something(field1: type, field2: type). This happened because the parser never parsed that syntax. This has been fixed.

    Now when you do:

    enum Test: Player(name: str, health: int) end func main(): Test return Player.Test( name: "Player 1", health: 100 ) end

    It generates correct code:

    typedef struct Test_Player_Payload { Slice_char name; int32_t health; } Test_Player_Payload; Test_Player_Payload Test_Player_Payload_new(Slice_char name, int32_t health); typedef enum { Test__Player = 0, } TestTag; typedef struct { TestTag tag; union { Test_Player_Payload Player; } data; } Test; int32_t vix_main(); Test_Player_Payload Test_Player_Payload_new(Slice_char name, int32_t health) { Test_Player_Payload instance; instance.name = name; instance.health = health; return instance; }

    Fourth Bug: Enum Constructor Compilation

    Meanwhile, while testing, we found another bug where the codegen wasn't compiling:

    return Player.Test( name: "Player 1", health: 100 )

    This bug has been fixed too.

    Fifth Bug: While Loop Generation

    The fifth bug is about while. When you created any while loop, it would not generate because codegen_enum was failing—it always generated label_0: without any loop at all. This problem has been fixed.

    Sixth Bug: Struct Implementation and Main Function

    The last bug: When you do something like:

    impl Test: func new(): Test Test( i = 0, x = "" ) end func test(&mut self, input: int): Result[int, str] if input != 0 then self.i = 10 return Ok(0) end Err("Failed: input is zero") end end func main() create i: int = 10 create test = Test create x = test.test(i) end

    This generated correct impl code, but everything inside func main() was generating incorrect IR:

    int32_t vix_main() { int32_t t0 = 10; const int32_t var_i = t0; test_test(var_i); int32_t t1 = 0; }

    This bug has been fixed and now it generates correct IR.
